# Desktop Release Pipeline

This document describes the signed desktop release flow introduced in PR4 and
extended in PR6 with Tauri updater support.

The desktop slice is only one part of the launch chain. For the full
API + DB + desktop promotion map, start with
[docs/RELEASE_TOPOLOGY.md](RELEASE_TOPOLOGY.md).

## Release Model

- Pull requests validate desktop code and packaging inputs.
- Pushes to `main` build internal desktop artifacts against the staging desktop
  channel.
- Pushing a version tag like `v0.1.0` triggers:
  - production database migration deploy
  - production API deploy
  - signed desktop release builds for macOS and Windows

The release tag is the manual promotion step. Once the tag is pushed, GitHub
Actions publishes the production desktop release automatically.

## Release Command

Create and push the release tag from the exact commit you want to ship:

```bash
git checkout main
git pull
git tag -a v0.1.0 -m "Release v0.1.0"
git push origin v0.1.0
```

The desktop release workflow verifies that:

- `desktop/package.json`
- `desktop/src-tauri/tauri.conf.json`
- `desktop/src-tauri/Cargo.toml`

all share the same version, and that the pushed tag matches `v<desktop-version>`.

## GitHub Actions Workflows

- `.github/workflows/desktop-build.yml`
  validates desktop JS on PRs and builds staging-channel desktop artifacts on
  `main`.
- `.github/workflows/desktop-release.yml`
  builds and publishes signed desktop installers on `v*` tags, uploads the
  updater signatures, and publishes `latest.json` for in-app updates.

## Production Secrets

Set these in the GitHub `production` environment used by the desktop release
workflow.

### Tauri updater signing

- `TAURI_UPDATER_PUBLIC_KEY`
  Public key generated by `tauri signer generate`. This is baked into the
  production desktop build so the app can verify updater artifacts.
- `TAURI_SIGNING_PRIVATE_KEY`
  Contents of the updater private key generated by `tauri signer generate`.
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
  Optional password for the updater private key. Leave unset if you generated
  the key without a password.

Generate the updater keypair once on a trusted machine:

```bash
cd desktop
pnpm tauri signer generate -w ~/.tauri/ultralight-updater.key
```

Store the private key somewhere durable before you put it in GitHub secrets.
If you lose it, already-installed desktop clients will no longer trust future
updates signed with a different key.

### macOS signing + notarization

- `APPLE_CERTIFICATE`
  Base64-encoded `.p12` Developer ID Application certificate.
- `APPLE_CERTIFICATE_PASSWORD`
  Password used when exporting the `.p12`.
- `KEYCHAIN_PASSWORD`
  Temporary keychain password for the macOS runner.
- `APPLE_API_ISSUER`
  App Store Connect API issuer ID.
- `APPLE_API_KEY`
  App Store Connect API key ID.
- `APPLE_API_KEY_P8`
  Contents of the downloaded `AuthKey_<KEY_ID>.p8` file.

The workflow imports the certificate, resolves the signing identity, writes the
App Store Connect key to disk, and notarizes the macOS app using the API key
credentials.

### Windows signing

The current workflow assumes an exportable `.pfx` certificate-based path.

- `WINDOWS_CERTIFICATE`
  Base64-encoded `.pfx` certificate.
- `WINDOWS_CERTIFICATE_PASSWORD`
  Password for the `.pfx`.

The workflow imports the certificate into the Windows certificate store,
extracts its SHA-1 thumbprint, and generates a temporary Tauri config with
`certificateThumbprint`, `digestAlgorithm = sha256`, and
`timestampUrl = http://timestamp.digicert.com`.

If you choose Azure Trusted Signing instead of a `.pfx` certificate, adapt the
Windows portion of the workflow rather than reusing these secrets directly.

## Updater Endpoint

Production desktop releases check GitHub Releases directly at:

```text
https://github.com/evrydayimruslin/ultralight/releases/latest/download/latest.json
```

The updater is enabled only in the tag-driven production release workflow. The
`main` desktop packaging job keeps building staging/internal artifacts without
requiring the updater signing key.

## Notes

- The workflow publishes release assets to the GitHub Release attached to the
  pushed tag.
- macOS release builds use the `darwin-universal` updater target so universal
  app bundles match the GitHub `latest.json` metadata.
- Windows updater metadata prefers the NSIS installer path, which matches the
  current-user installation mode already configured in `tauri.conf.json`.
- The current staging runtime shares production Cloudflare R2/KV bindings as an
  intentional interim step. Desktop production releases remain isolated by
  their production API and production Supabase configuration. This exception is
  part of the canonical isolation inventory in
  [docs/ENVIRONMENT_ISOLATION_MATRIX.md](ENVIRONMENT_ISOLATION_MATRIX.md),
  with owner, risk, and removal target tracked there. Keep it visible in
  release decisions until full Cloudflare-side isolation exists.

## Recovery Notes

- Treat production desktop release incidents as fix-forward by default: prefer a
  new patch release over mutating an existing tag.
- If users need an immediate fallback, point them to the prior signed GitHub
  Release artifact while the patch is prepared.
- Record whether the incident affects:
  - fresh installers
  - in-app updater flow
  - both
- Use
  [docs/ROLLBACK_REHEARSAL.md](ROLLBACK_REHEARSAL.md)
  for the rehearsal and release-day decision model.
