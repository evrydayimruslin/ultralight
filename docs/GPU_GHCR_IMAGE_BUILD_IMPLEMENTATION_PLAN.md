# GPU GHCR Image Build Implementation Plan

## Goal

Move GPU dependency setup out of the end-user cold-start path and into a developer-paid build step.

Today, GPU workers download app code from R2 and install `requirements.txt` when a fresh Runpod worker starts. That makes cold starts slower, more failure-prone, and more expensive for callers. The new path builds a per-app Docker image once, pushes it to GHCR, and points Runpod at that image.

## Locked Decisions

- Build runner: GitHub Actions.
- Image registry: GitHub Container Registry.
- Dockerfiles: generated by Galactic only. Developer-uploaded Dockerfiles are rejected or ignored for GPU apps in v1.
- Developer authoring surface: Tool Maker creates `main.py`, `requirements.txt`, `ultralight.gpu.yaml`, and `test_fixture.json`; the platform creates the Dockerfile.
- Base image profiles: `python-cuda` and `torch-cuda`.
- Default base profile: `python-cuda`.
- Tool Maker should choose `torch-cuda` when the app uses PyTorch, model inference, tensors, diffusion, embeddings, or otherwise has `torch`/`torchvision`/`torchaudio` dependencies.
- Build cost: passed through to the app owner in Light.
- Storage accounting: count app-specific source, build logs, and app-specific image layers toward the app owner's storage. Shared base layers are platform-owned and not counted per developer.
- Deletion: deleting an app/version deletes its source bundle, build artifacts, and GHCR image tag/digest. A background sweeper handles orphan cleanup.

## Product Behavior

Developer uploads a GPU app exactly as before:

```yaml
runtime: gpu
gpu_type: A40
python: "3.11"
base: torch-cuda
max_duration_ms: 30000
```

Upload returns immediately with `gpu_status: "building"`. During build, the platform:

1. Validates GPU config.
2. Estimates and holds build Light from the app owner.
3. Creates a Docker build context from uploaded files plus the Galactic harness.
4. Dispatches a GitHub Actions workflow.
5. GitHub Actions builds and pushes the image to GHCR.
6. The API creates a Runpod template/endpoint using that image.
7. Existing benchmark flow runs.
8. App becomes `live`.

End users still call apps the same way. The improvement is that fresh workers no longer install dependencies at startup.

### GitHub Actions Secrets

- `GPU_BUILD_CALLBACK_SECRET`

The workflow should use the built-in `GITHUB_TOKEN` with `packages: write` permission to push app images to GHCR. Avoid a broad long-lived GHCR write token if the workflow repository can own the package namespace.

### Runpod

- Saved container registry auth for GHCR private image pulls. Store the GHCR read username/token in Runpod once, then set `RUNPOD_CONTAINER_REGISTRY_AUTH_ID` to that saved registry credential id. The read token should only be able to pull GPU app images and shared base images.

## Build Pipeline Changes

### New Service

Create `api/services/gpu/image-builder.ts`.

Responsibilities:

- Generate Dockerfile from a locked template.
- Create build context metadata.
- Dispatch GitHub Actions workflow.
- Poll or receive callback for workflow status.
- Fetch logs or store workflow URL.
- Return image ref, digest, image size, duration, and build cost.

### Build Context Delivery

The API should not pass source files directly in the workflow dispatch payload. Instead:

1. Create a tarball build context:
   - generated `Dockerfile`
   - normalized app source files
   - generated `manifest.json`
   - platform `harness.py`
   - metadata file with `app_id`, `version`, `base`, and target image tag
2. Store the tarball in R2 under a short-lived build path.
3. Generate a one-time signed download URL or expose an authenticated internal download endpoint.
4. Dispatch the GitHub Actions workflow with:
   - `app_id`
   - `version`
   - `context_url`
   - `target_image`
   - `callback_url`
5. GitHub Actions downloads the tarball, builds it, pushes the image, then calls back with status, digest, size, and logs URL.

If signed R2 URLs are not already available, add an internal build-context endpoint authenticated by `GPU_BUILD_CALLBACK_SECRET` or a separate short-lived token.

### Dockerfile Template

Generated shape:

```dockerfile
FROM ghcr.io/ultralight/gpu-base-torch-cuda:py3.11-cuda12

WORKDIR /app

COPY requirements.txt /app/requirements.txt
RUN if [ -f /app/requirements.txt ]; then pip install --no-cache-dir -r /app/requirements.txt; fi

COPY . /app
COPY harness.py /app/harness.py

ENV ULTRALIGHT_BAKED_IMAGE=1

CMD ["python", "-u", "/app/harness.py"]
```

Rules:

- No developer Dockerfile.
- No custom entrypoint.
- No secret values in build args.
- Requirements install at build time.
- Use pinned base images by digest where possible.

### Existing File Changes

- `api/services/gpu/builder.ts`
  - Replace reference-only Dockerfile storage with real image build dispatch.
  - Store source bundle/build artifacts as today.
  - After image build success, call provider with `imageRef`.
  - Charge/settle build cost against app owner.

- `api/services/gpu/provider.ts`
  - Add `imageRef?: string` and `imageDigest?: string` to `BuildContainerParams` or introduce `CreateEndpointParams`.

- `api/services/gpu/runpod.ts`
  - Use per-app image ref when creating the Runpod template.
  - Keep shared template fallback only behind explicit legacy flag.
  - Store template env only for platform metadata, not code download.

- `api/services/gpu/harness.py`
  - If `ULTRALIGHT_BAKED_IMAGE=1`, skip `download_code()`.
  - Remove runtime `pip install` from the normal path.
  - Keep R2 download path only as legacy fallback until migration is complete.

- `api/services/gpu/test-template/Dockerfile`
  - Update to match new base/harness assumptions or rename as legacy template.

## Billing Changes

Add build preflight:

```text
estimated_build_light =
  ceil(estimated_minutes) * GPU_BUILD_LIGHT_PER_MINUTE
```

For GitHub Linux 2-core hosted runner pricing, the current external rate is roughly `$0.006/min`, so at 100 Light per dollar:

```text
GPU_BUILD_LIGHT_PER_MINUTE = 0.6
```

Suggested beta defaults:

- `GPU_BUILD_LIGHT_PER_MINUTE=1`
- `GPU_BUILD_ESTIMATE_MINUTES_DEFAULT=10`
- `GPU_BUILD_MAX_ESTIMATE_MINUTES=30`

This makes the hold conservative but still small. On completion, settle actual minutes and release any excess. If the build fails due to platform/GitHub/Runpod infra, refund. If it fails due to invalid developer dependencies, charge actual build cost but no runtime execution cost.

## Storage Accounting

Count these against owner storage:

- uploaded GPU source files
- generated manifest
- generated Dockerfile
- build logs
- app-specific image layer bytes

Do not count:

- shared `python-cuda` base image layers
- shared `torch-cuda` base image layers
- Runpod provider-side image cache

Implementation options:

1. MVP: record compressed final image size as `gpu_image_user_storage_bytes`, minus known base image size.
2. Better: inspect OCI manifest layers and subtract layer digests that belong to the selected base image.

Use option 1 for launch if OCI layer accounting takes too long. It slightly overcharges storage for shared dependency layers, but avoids blocking the build-pipeline improvement.

## Open Questions

These are the remaining decisions before implementation:

1. Which GitHub repository owns the build workflow?
2. Which registry namespace should GHCR use?
3. Should app images be private only, or public for open-source/public GPU apps later?
4. Should v1 block unpinned requirements or only warn?
5. Should failed developer-dependency builds charge actual build cost immediately, or be free during beta?
6. Should storage accounting launch with final-size-minus-base approximation or exact OCI layer accounting?
7. Should workflow dispatch use a GitHub App or a fine-grained PAT?

Recommended answers for v1:

1. Use a private `ultralight-gpu-builds` repo under the Galactic org.
2. Use `ghcr.io/ultralight/gpu-apps`.
3. Keep all app images private.
4. Warn on unpinned requirements at first; block later for published apps.
5. Charge actual build cost after beta starts; during internal alpha, absorb it.
6. Start with approximation; move to exact layer accounting once build flow is stable.
7. Use a GitHub App for production. A fine-grained PAT is acceptable for a short internal alpha if speed matters more than perfect token hygiene.
